# ADR-0032: YAML-LD ontology and pySHACL data quality reports

**Status:** Accepted  
**Date:** 2026-07-09  
**Deciders:** cognite-fholm  
**Related:** [ADR-0022](./0022-yaml-ld-interconnected-data.md), [ADR-0023](./0023-shacl-neo4j-projection-no-fuseki.md), [AI-sailing-data DATA_QUALITY.md](https://github.com/cognite-fholm/AI-sailing-data/blob/main/docs/DATA_QUALITY.md)

---

## Context

AI-sailing-data uses YAML-LD facts and shore pySHACL validation ([ADR-0023](./0023-shacl-neo4j-projection-no-fuseki.md)), but crews and agents lacked:

- A single **ontology** document explaining how boats, races, and fleets relate for reasoning
- **Scoped data quality rules** for own boat, competitors, and races
- **Actionable DQ reports** per change — not only pass/fail CI

---

## Decision

### 1. YAML-LD ontology

Publish `schema/yaml-ld/ontology.yamlld` in AI-sailing-data:

- Classes: `Boat`, `Race`, `Fleet`, `FleetParticipation`, `OrcCertificate`, …
- Properties: `competesIn`, `hasFleet`, `hasCertificate`, …
- `reasoning_patterns` for own boat, competitor, and race participation

Align with **yaml-ld-read** skill rules (D13–D15).

### 2. Fleet participation map

Each race folder MAY include `planning/fleet-participation.yaml` (`kind: FleetParticipation`):

- `spec.race` and `spec.fleet` — document `@id` node objects
- `spec.entrants[]` — `sail_number`, `role`, optional `boat` link to `boats/{sail}/boat.yaml`

Generated by `scripts/sync_fleet_participation.py` from `fleet.yaml`.

### 3. pySHACL data quality shapes

Add `schema/shacl/data-quality.shacl.ttl`:

| Shape | Scope |
|-------|--------|
| `OwnBoatDocumentShape` | Own boat identity |
| `CompetitorBoatDocumentShape` | Competitor boat stubs |
| `RaceDataQualityShape` | Race manifest |
| `FleetDataQualityShape` | Fleet entrant list |
| `FleetParticipationShape` | Boats ↔ race YAML-LD links |

Cross-file checks (boat stub exists, polar present, race folder completeness) complement SHACL in `scripts/generate_dq_report.py`.

### 4. DQ report folder

On CI and locally, `scripts/generate_dq_report.py` writes:

```
dq-report/
  summary.md / summary.json
  boats/{sail_number}.md
  races/{regatta_id}.md
```

CI fails on **error**-level violations; warnings (e.g. pending competitor enrichment) are reported.

---

## Consequences

### Positive

- Agents reason over linked data with explicit ontology + skill alignment
- Own boat, fleet, and race issues surface in scoped reports
- `FleetParticipation` makes “who competes in what race” machine-readable

### Trade-offs

- DQ reports must be regenerated and committed on `main` when facts change
- Competitor boat warnings remain until Phase 4 enrichment completes
